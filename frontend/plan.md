### Typography

**Heading Font:** Serif (appears to be a serif font family like Playfair Display or similar)

**Body Font:** Sans-serif (appears to be clean sans-serif)

**Buttons:**

- Solid buttons: Olive/gold background, white text, no border

- Button states: Normal, hover (likely darkened), active
- Icons sometimes included (arrows, etc.)

**Cards:**

- Product cards: Image + text info below
- Benefit cards: Icon + text
- Value cards: Icon + heading + description
- Light background or shadow for elevation

**Inputs & Forms:**

- Text inputs: Light background, subtle borders
- Placeholders: Visible and styled
- Labels: Serif headings for form sections
- Focus states: Not visible in static design

**Navigation:**

- Horizontal menu in header
- Icon buttons (theme toggle, cart, user)
- Responsive menu likely hamburger on mobile

**Icons:**

- Circular icons for benefits
- Navigation icons (search, cart, user, sun/moon)
- Plus/minus for FAQ accordion
- Arrow icons on buttons
- Simple, minimal style

# SECTION 1: PROJECT ARCHITECTURE

## 🏗️ Framework & Technology Stack

### Frontend Framework: **React 18**

**Rationale:**

- Component-based architecture aligns with reusable UI components identified
- Large ecosystem and community support
- Excellent for building complex e-commerce interfaces
- Good performance optimization capabilities
- Rich tooling and library support
- Perfect for managing dynamic state (filters, cart, theme toggle)

### Styling: **Tailwind CSS**

**Rationale:**

- Utility-first approach enables rapid, consistent styling
- Built-in responsive breakpoint system matches our design needs
- Theme customization allows easy dark/light mode implementation
- Smaller bundle size compared to traditional CSS frameworks
- Excellent for maintaining design consistency across components
- JIT compilation ensures we only ship used styles
- Color palette and spacing can be centralized in config

**Tailwind Configuration Focus:**

- Custom color variables for primary (olive/gold), secondary, and theme colors
- Extended typography scale for serif and sans-serif families
- Custom spacing system aligned with design
- Dark mode support built-in via class strategy

### State Management: **Context API + React Hooks (with Zustand as alternative)**

**Primary Approach: Context API + Hooks**

- Simple enough for the project scope
- No external dependencies needed for basic state
- Sufficient for managing: theme toggle, cart items, user session, filters

**Additional State Management (Zustand)**

- Use for more complex states if Context API becomes unwieldy
- Lighter weight than Redux, easier to learn
- Minimal boilerplate compared to Redux
- Good for: global notifications, UI state, complex filter state

**State Contexts Needed:**

- ThemeContext (light/dark mode toggle)
- CartContext (shopping cart items, quantities)
- AuthContext (user login, session, admin access)
- FilterContext (shop page filters, applied filters)
- NotificationContext (toast messages, alerts)

### Build Tool: **Vite**

**Rationale:**

- Fast development server (near-instant HMR)
- Optimized production builds
- ESM-first approach
- Significantly faster than Create React App
- Modern tooling for React projects

### UI Component Library: **Headless (Custom Components)**

**Rationale:**

- Full control over design matching luxury aesthetic
- No pre-designed component limitations
- Can build components incrementally
- Use Tailwind + custom React components
- Alternative: Radix UI (for unstyled, accessible components if needed later)

## 🎯 DEVELOPMENT WORKFLOW

**Component Organization Philosophy:**

- `components/common/` - Reusable, generic components (Button, Card, Modal, etc.)
- `components/layout/` - Page layout wrappers (MainLayout, AdminLayout)
- `components/features/` - Feature-specific components organized by page
- `pages/` - Page containers that combine layouts and features
- `hooks/` - Custom React hooks for common logic
- `context/` - Global state management
- `utils/` - Pure functions and helpers
- `data/` - Mock data for development

**Component Pattern:**

- Functional components with hooks
- Props for customization
- Destructuring for clarity
- Proper TypeScript-ready structure (even if not using TS initially)

**Styling Pattern:**

- Tailwind utility classes in className
- CSS modules only if needed for complex styling
- CSS variables for theme colors in root
- No inline styles unless absolutely necessary
